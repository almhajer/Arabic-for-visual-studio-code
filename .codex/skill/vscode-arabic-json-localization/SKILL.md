# SKILL.md

name: vscode-arabic-json-localization

description: >
  مهارة مخصصة لتعريب ملفات JSON الخاصة ببرنامج VS Code وإضافاته من
  الفرنسية إلى العربية، مع الحفاظ الكامل على بنية JSON، وعدم تغيير
  المفاتيح أو placeholders أو المتغيرات أو التعابير الخاصة، واعتماد
  ترجمة عربية طبيعية ومهنية ومتسقة مع TERMINOLOGY.md.

## الهدف
هذه المهارة مخصصة لتنفيذ ترجمة احترافية لملفات JSON في مشروع تعريب VS Code، بحيث تكون النتيجة:
- عربية سليمة وطبيعية
- دقيقة معنويًا
- مناسبة لواجهة المستخدم
- متسقة في كامل المشروع
- آمنة من ناحية بنية JSON
- متوافقة مع ملف TERMINOLOGY.md

## الملفات المستهدفة
تشمل هذه المهارة الملفات التالية:
- `main.i18n.json`
- `package.json`
- `extensions/*.i18n.json`

## متى تستخدم هذه المهارة
استخدم هذه المهارة عندما تكون المهمة:
- ترجمة ملف JSON من الفرنسية إلى العربية
- مراجعة ترجمة موجودة وتحسينها
- إصلاح كلمات غير مترجمة أو مختلطة
- توحيد المصطلحات التقنية
- إزالة البقايا الإنجليزية والفرنسية
- تحسين صياغة الواجهة
- مراجعة الاتساق الاصطلاحي بين الملفات

## المرجع الاصطلاحي الإلزامي
قبل تنفيذ أي ترجمة أو مراجعة:
- يجب الرجوع إلى `TERMINOLOGY.md`
- يجب الالتزام بالمصطلحات المعتمدة فيه
- عند وجود أكثر من ترجمة ممكنة، تُختار الصيغة الأقرب إلى:
  1. وضوح المعنى
  2. طبيعية العربية
  3. اتساق المشروع
  4. أسلوب واجهات VS Code

## النتيجة المطلوبة
المخرج النهائي يجب أن يحقق الشروط التالية:
- JSON سليم 100٪
- المفاتيح لم تتغير
- placeholders لم تتغير
- النصوص التقنية الداخلية لم تُترجم بالخطأ
- النصوص الظاهرة للمستخدم مترجمة ترجمة طبيعية
- لا توجد بقايا فرنسية أو إنجليزية بلا سبب
- لا توجد ترجمة حرفية ركيكة
- المصطلحات موحدة عبر كامل المشروع

## قواعد صارمة

### 1) لا تغيّر بنية JSON
ممنوع:
- تغيير المفاتيح
- حذف أو إضافة أقواس
- حذف أو إضافة فواصل
- كسر السلاسل النصية
- تغيير نوع القيم
- إعادة هيكلة الملف
- حذف حقول أو إضافة حقول دون طلب صريح

### 2) ترجم القيم النصية الظاهرة للمستخدم فقط
ترجم فقط النصوص التي يراها المستخدم النهائي، مثل:
- العناوين
- الأوصاف
- رسائل الخطأ
- رسائل الحالة
- أسماء الأوامر الظاهرة
- عناصر القوائم
- التلميحات
- النصوص المساعدة

### 3) لا تترجم العناصر التقنية الداخلية
اتركها كما هي:
- مفاتيح JSON
- command ids
- configuration keys
- property names
- identifiers
- enum values التقنية
- internal ids
- package names
- extension ids
- file paths
- URLs البرمجية إن كانت قيمة تقنية داخلية غير معروضة للمستخدم

### 4) لا تلمس placeholders أو الرموز الخاصة
اتركها كما هي حرفيًا:
- `{0}`, `{1}`, `{2}`
- `{name}`, `{value}`, `{label}`
- `${variable}`
- `%s`, `%d`, `%f`
- `{{value}}`
- `\n`, `\t`, `\r`
- HTML tags
- Markdown syntax
- النصوص البرمجية داخل backticks
- التعابير المنتظمة أو الأمثلة البرمجية ما لم تكن العبارة نفسها موجهة للمستخدم

## مبدأ الترجمة
الترجمة هنا Localization وليست ترجمة حرفية.

هذا يعني:
- ترجم المعنى لا الكلمات
- اجعل النص مناسبًا لواجهة المستخدم
- اجعل الجملة عربية وكأنها أصلية
- إذا كانت الحرفية تضعف الوضوح، اختر الصياغة العربية الأوضح
- إذا كان المصطلح تقنيًا مشهورًا، يمكن إبقاؤه ضمن جملة عربية سليمة وفق `TERMINOLOGY.md`

## منهج اتخاذ القرار
عند وجود أكثر من ترجمة ممكنة، اعتمد هذا الترتيب:
1. سلامة JSON
2. الحفاظ على placeholders والمتغيرات
3. صحة المعنى
4. وضوح العبارة للمستخدم
5. الاتساق الاصطلاحي
6. الإيجاز المناسب

## قواعد الصياغة العربية

### الأزرار وعناصر القوائم
يجب أن تكون:
- قصيرة
- مباشرة
- واضحة

أمثلة:
- Open → فتح
- Save → حفظ
- Close → إغلاق
- Retry → إعادة المحاولة
- Cancel → إلغاء

### أوصاف الإعدادات
يفضل أن تكون:
- تفسيرية
- دقيقة
- طبيعية

أمثلة:
- Controls whether ... → يحدد ما إذا كان ...
- Specifies ... → يحدد ...
- Enables ... → يمكّن ...
- The maximum number of ... → الحد الأقصى لـ ...

### رسائل الخطأ
يجب أن تكون:
- مباشرة
- واضحة
- غير حرفية

أمثلة:
- Failed to load configuration → فشل تحميل الإعدادات
- Cannot open file → تعذر فتح الملف

### الرسائل التوضيحية
اجعلها:
- سهلة الفهم
- غير متكلفة
- مناسبة لسياق المطورين والمستخدمين

## المصطلحات التقنية
تعامل مع المصطلحات التقنية وفق `TERMINOLOGY.md` فقط.

### قواعد سريعة
- المصطلح الواجهي العام يُترجم للعربية
- اسم التقنية المشهور يمكن إبقاؤه
- الاختصار المشهور يمكن إبقاؤه إذا كانت العبارة العربية واضحة
- لا تترك كلمة إنجليزية أو فرنسية داخل النص الظاهر للمستخدم دون مبرر
- البقايا الفرنسية يجب ترجمتها بالكامل

## مفاتيح لوحة المفاتيح
التزم بالقواعد الواردة في `TERMINOLOGY.md`.

### القاعدة العامة
- أسماء المفاتيح الفعلية تبقى غالبًا كما هي
- تُترجم الجملة المحيطة بالعربية
- تُستخدم الصيغة الوصفية عند الحاجة

### أمثلة
- Press Enter to continue → اضغط Enter للمتابعة
- Press Ctrl+C to copy → اضغط Ctrl+C للنسخ
- Use the arrow keys to navigate → استخدم مفاتيح الأسهم للتنقل
- Keyboard shortcut → اختصار لوحة المفاتيح
- Keybinding → ربط المفاتيح

## قواعد خاصة بالملفات

### أولًا: `package.json`
في هذا الملف:
- ترجم فقط النصوص الظاهرة للمستخدم
- لا تترجم الحقول التقنية

#### ترجم مثلًا:
- `displayName`
- `description`
- `title`
- `command` titles الظاهرة
- `category` الظاهرة
- `configuration.*.description`
- `walkthrough` titles/descriptions إذا كانت للمستخدم

#### لا تترجم:
- `name`
- `publisher`
- `version`
- `engines`
- `scripts`
- `activationEvents`
- `command ids`
- `contributes` keys
- `configuration` property names

### ثانيًا: `main.i18n.json`
في هذا الملف:
- اجعل الترجمة مختصرة وواضحة
- راعِ أن كثيرًا من النصوص متكرر أو مترابط
- وحّد المصطلحات عبر كامل الملف
- راجع العبارات الحساسة مثل:
  - editor
  - settings
  - diff
  - preview
  - focus
  - selection
  - accessibility
  - configuration
  - notebook
  - terminal

### ثالثًا: `extensions/*.i18n.json`
في هذه الملفات:
- حافظ على نفس القاموس المعتمد في `main.i18n.json`
- لا تجعل الامتداد يغيّر مصطلحًا أساسيًا تم تثبيته سابقًا
- أسماء اللغات أو الأدوات أو المنتجات تبقى حسب ما يحدده `TERMINOLOGY.md`
- إذا ظهر مصطلح جديد غير موجود في القاموس، اختر صياغة متسقة وأضفه لاحقًا إلى `TERMINOLOGY.md`

## استراتيجية العمل للملفات الكبيرة
إذا كان الملف كبيرًا جدًا:
- لا تترجمه دفعة واحدة
- اعمل على دفعات صغيرة أو متوسطة
- الأفضل: 500 إلى 1500 سطر
- أو حسب الأقسام المنطقية

### لكل دفعة
اتبع هذا التسلسل:
1. اقرأ الدفعة كاملة
2. حدّد النصوص الظاهرة للمستخدم فقط
3. ترجمها ترجمة طبيعية
4. افحص placeholders
5. افحص البنية
6. راجع المصطلحات
7. قارنها بما سبق
8. أخرج النتيجة النهائية

### بعد انتهاء الملف كاملًا
يجب تنفيذ مراجعة شاملة للتأكد من:
- توحيد المصطلحات
- إزالة الترجمة الحرفية
- إزالة البقايا الإنجليزية والفرنسية
- الحفاظ على صحة JSON
- بقاء placeholders كما هي

## مراجعة الجودة

### ترجمة ممتازة
- عربية طبيعية
- دقيقة
- مناسبة للواجهة
- متسقة
- خالية من البقايا الأجنبية غير المبررة
- تحافظ على المعنى دون حرفية مزعجة

### ترجمة مرفوضة
- حرفية
- ركيكة
- مختلطة عربي/إنجليزي بلا داعٍ
- مختلطة عربي/فرنسي
- تكسر placeholders
- تكسر JSON
- تترجم identifiers التقنية
- تختلف عن القاموس الاصطلاحي المعتمد

## خطوات المراجعة النهائية
عند مراجعة ملف مترجم:
1. افحص جودة العربية
2. افحص الاتساق الاصطلاحي
3. افحص البقايا الإنجليزية والفرنسية
4. افحص المصطلحات التقنية والاختصارات
5. افحص مفاتيح الكيبورد إن وجدت
6. افحص placeholders
7. افحص صحة JSON
8. عدّل فقط المواضع التي تحتاج تحسينًا

## شكل الإخراج

### عند تنفيذ ترجمة
- أخرج JSON فقط
- لا تضف شرحًا
- لا تضف Markdown
- لا تضف أي تعليق خارجي
- لا تغيّر النصوص الصحيحة دون داعٍ

### عند تنفيذ مراجعة
- عدّل فقط المواضع التي تحتاج تحسينًا
- لا تعِد كتابة الملف كاملًا إن لم يكن ذلك مطلوبًا
- لا تلمس النصوص الصحيحة
- بعد الانتهاء يمكن إعطاء موجز مختصر جدًا عن:
  - عدد المواضع المصححة
  - بقاء JSON سليمًا
  - وجود مواضع مشكوك فيها أو عدمه
